1. Introduction

Git is a distributed version control system for tracking changes in files. GitHub is an online platform that hosts Git repositories, providing collaboration features like pull requests, issues, and project tracking.

With Git, developers can:

  • Track changes to code over time
  • Collaborate with others through branches and merges
  • Recover previous versions of a project

2. Git Basics

2.1 Repositories

A Git repository stores project history and metadata in a .git folder. It can be local on your machine or remote on a server.

2.2 Initializing a Repository

git init

Creates a new Git repository in the current folder.

2.3 Checking Status

git status

Shows modified, staged, and untracked files.

2.4 Adding and Committing

git add .
git commit -m "Initial commit"

git add stages changes, and git commit records them with a message.

2.5 Viewing History

git log

Displays commit history. Add --oneline for a shorter view.

3. GitHub Overview

3.1 What GitHub Provides

  • Remote repository hosting
  • Branch management and pull requests
  • Issue tracking and project boards
  • CI/CD integrations and code reviews

3.2 Connecting Local Git to GitHub

git remote add origin https://github.com/username/repo.git
 git push -u origin main

Use SSH if you prefer secure keys:

git remote add origin git@github.com:username/repo.git

4. Branching & Merging

Branches allow parallel work without affecting the main line of development.

Create a branch

git branch feature/login

Switch branches

git checkout feature/login

Create and switch in one step

git switch -c feature/login

Merge a branch

git checkout main
git merge feature/login

If there are conflicts, Git marks the files and you resolve them manually.

5. Pull Requests

Pull requests are GitHub’s review mechanism for proposed changes. They let maintainers inspect code, comment, and merge when ready.

Key Pull Request steps

  1. Push your branch to GitHub
  2. Open a pull request from the branch to the target branch
  3. Describe the change clearly
  4. Respond to review comments
  5. Merge once approved

Use draft pull requests for work in progress, and keep each PR focused on one issue.

6. Advanced Git

6.1 Rebasing

Rebase applies commits from one branch onto another, creating a cleaner history.

git checkout feature/login
git rebase main

Use rebasing carefully on local branches. Avoid rebasing shared history.

6.2 Interactive Rebase

git rebase -i HEAD~4

Rewrite commits to squash, reorder, or edit commit messages.

6.3 Cherry-pick

git cherry-pick a1b2c3d

Apply a specific commit from another branch into the current branch.

6.4 Stashing

git stash
 git stash pop

Temporarily save incomplete work, switch context, and restore it later.

6.5 Resetting

Three reset modes:

  • git reset --soft: keep changes staged
  • git reset --mixed: keep working tree changes unstaged
  • git reset --hard: discard all changes

6.6 Tags

git tag -a v1.0 -m "Release 1.0"
 git push origin v1.0

Tags mark releases or important points in history.

7. Common Workflows

7.1 Feature Branch Workflow

Use a branch per feature, merge into main once complete.

7.2 GitHub Flow

  1. Create a branch
  2. Commit changes
  3. Open a pull request
  4. Review and merge

7.3 Git Flow

A structured approach with develop, release, and hotfix branches.

7.4 Fork & Pull Model

Used in open source: contributors fork the repo, push branches to their fork, then create pull requests to the upstream repository.

8. Best Practices

  • Write clear commit messages
  • Keep commits small and focused
  • Use branches for features and fixes
  • Pull remote changes before starting work
  • Review diffs before committing and pushing
  • Use .gitignore to exclude build artifacts and secrets
